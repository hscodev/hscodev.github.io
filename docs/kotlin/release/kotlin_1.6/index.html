Kotlin 1.6 릴리즈

요약 : 링크

릴리즈 노트 : 링크


Language features​

Kotlin 1.5.30 기반으로 기능 안정화.


1. Sealed (exhaustive) when statements​

Sealed when은 코틀린 컴파일러가 when 문장이 완벽하지 않은 경우 경고해줍니다. 이것은 코드를 더 안전하게 만듭니다.

코틀린은 sealed 클래스나, enum, Boolean 타입의 when 문법에 대해 항상 정확한 검사를 해왔습니다. 이것은 대수적 데이터 유형으로 도메인을 설계할 때 유용합니다. 예를 들어 sealed 클래스를 이용하여 각 다른 연락 수단을 설계한 경우입니다.

sealed class Contact {
   data class PhoneCall(val number: String) : Contact()
   data class TextMessage(val number: String) : Contact()
   data class InstantMessage(val type: IMType, val user: String) : Contact()
}

여기서 연락 수단별로 각 다른 결과를 반환하는 when 구문을 작성했을 때, 모든 유형을 제어하지 않으면 컴파일러는 오류를 나타내게 됩니다.

fun Rates.computeMessageCost(contact: Contact): Cost =
   when (contact) { // ERROR: 'when' expression must be exhaustive
       is Contact.PhoneCall -> phoneCallCost
       is Contact.TextMessage -> textMessageCost
   }

이러한 기능은 코드를 작성하거나 유지보수 하는데에 매우 도움이 됩니다. 만약 나중에 또다른 연락 방식이 추가되면 어플리케이션 전역적으로 검사하여 제어를 추가해 주도록 도와줍니다.

그러나 코틀린 1.6 이전에 아래와 같이 반환 함수가 아닌 처리 함수인 경우 오류 없이 성공적으로 컴파일 되었었습니다.

fun sendAnnouncement(contact: Contact, announcement: Announcement) {
   when (contact) {
       is Contact.PhoneCall -> schedulePhoneCall(contact.number, announcement)
       is Contact.TextMessage -> sendTextMessage(contact.number, announcement)
   }
}

컴파일러의 메세지 없이 IDE의 약한 Inspection만 노출하고 있었습니다. 코틀린 1.6부터 다음과 같은 컴파일러 경고를 노출하게 됩니다.

Non-exhaustive 'when' statements on sealed class/interface will be prohibited in 1.7. Add an 'is InstantMessage' branch or 'else' branch instead.

코틀린 1.7부터 이것은 오류로 간주됩니다. 이것과 관련해 변경 사유와 효과에 대해 더 자세히 알고싶은 경우 다음 링크를 확인하세요. KT-47709


2. Suspending functions as supertypes​

코틀린 1.6에서는 super interface로서 suspend 함수형을 구현하기 위한 지원을 안정화 했습니다. 이것은 코틀린 코루틴 디자인에 있어서 잃어버린 한 조각이었습니다.

코틀린 API를 설계할 때 다른 라이브러리 함수의 행동을 커스텀하고 싶은 경우 함수형을 받아드리는 것은 관용적입니다.

예를 들어, kotlinx.coroutines API에는 이와 유사한 멤버 함수가 Job 인터페이스에 있습니다.

fun invokeOnCompletion(handler: () -> Unit)

실제 코드에서 invokeOnCompletion{ doSomething() } 처럼 람다를 이용하여 편리하게 사용할 수 있습니다. completion을 처리하려는 클래스가 있는 경우 추가 람다를 만들지 않고 () -> Unit을 클래스에 직접 구현하여 코드를 최적화할 수 있습니다.

class MyCompletionHandler : () -> Unit {
   override fun invoke() { doSomething() }
}

코틀린 1.6부터 suspend 선언이 가능합니다.

public fun launchOnClick(action: suspend () -> Unit) {}

그리고 람다 함수를 전달하거나 중단 함수를 참조하는 것은 이제 제한이 없습니다. 추가적으로 클래스로도 동일하게 중단함수를 적용할 수 있습니다.

class MyClickAction : suspend () -> Unit {
   override suspend fun invoke() { doSomething() }
}

3. Suspend conversions​

코틀린 1.6은 suspend 함수와 일반 함수의 변환을 안정화합니다. 이제 매개 변수로 suspend가 예상되는 부분에 일반적인 함수를 전달할 수 있습니다. 컴파일러가 자동으로 변환을 수행합니다.

이것은 코틀린의 일반 함수와 suspend 함수 사이의 상호 작용에서의 작은 불일치를 수정했습니다. 코틀린 flow의 collect 호출처럼 suspend 함수에서 람다를 이용하여 고차 함수를 사용할 수 있습니다.

flow.collect { processItem(it) }

fun processItem(item: Item) { /* ... */ }

일반 함수에 대한 참조를 전달하는 것이 편리할 수 있습니다.

flow.collect(::processItem)

함수에 대한 참조를 변수로 추출하여 코드의 동작을 사용자 지정할 수 있습니다. 코트린 1.6 이전 버전에서는 작동하지 않았습니다. 이는 suspend 유형의 매개변수로 전달되는 함수가 이미 있기 때문입니다.

val processingFunction = ::processItem
flow.collect(processingFunction) // ERROR: Type mismatch

코틀린 1.6에서는 위의 코드도 컴파일되고 잘 동작합니다.


4. Improved type inference for recursive generic types​

1.6.0부터, 코틀린 컴파일러는 재귀적 제네릭이라면 대응하는 타입 매개변수에 기반하여 타입 인수를 유추할 수 있습니다. 이를 통해 자바에서 빌더 API를 만드는 데 자주 사용되는 재귀적인 제네릭 타입으로 다양한 패턴을 만들 수 있습니다.

// Before 1.5.30
val containerA = PostgreSQLContainer<Nothing>(DockerImageName.parse("postgres:13-alpine")).apply {
    withDatabaseName("db")
    withUsername("user")
    withPassword("password")
    withInitScript("sql/schema.sql")
}

// With compiler option in 1.5.30 or by default starting with 1.6.0
val containerB = PostgreSQLContainer(DockerImageName.parse("postgres:13-alpine"))
    .withDatabaseName("db")
    .withUsername("user")
    .withPassword("password")
    .withInitScript("sql/schema.sql")

5. Builder inference improvements​

코틀린 1.5.3에서는 빌더 호출에 대한 타입 정보를 람다 내부에서 가져올 수 있게 하는 -Xunrestrict-builder-inference 컴파일러 옵션을 도입하였습니다. 이것은 buildList() 의 람다 안에서 get()과 같이 아직 유추되지 않은 호출을 만들 수 있게 합니다.

1.6.0부터는 제한된 함수 호출을 위해 -Xunrestricted-builder-inference을 지정할 필요가 없습니다. -Xenable-Builder-Inference 컴파일러 옵션을 사용하면 @BuilderInference 주석을 적용하지 않고도 일반 빌더를 작성할 수 있으며, 타입 추론이 타입 정보를 판단할 수 없는 경우 빌더 추론을 자동으로 활성화합니다.


Standard library​


1. New functions for standard input​

Kotlin 1.6.0에서는 표준 입력에서 한 줄을 읽은 후 not-null assertion 연산자(!!)를 사용하여 초보자의 경험을 개선하고 Kotlin을 가르치는 것을 단순화할 필요가 없어졌습니다.

콘솔에서 값을 읽을 때 사용할 수 있는 새로운 기능을 제공합니다.

  • readIn() 함수는 EOF에 다다르면 예외를 발생합니다. readLine()의 !!를 사용하여 null 체크하는 대신에 이 함수를 사용하세요.
  • readInOrNull() 함수는 null 반환의 대안입니다. readLine() 함수와 동작 방식이 동일하지만 조금 더 대표적인 이름입니다.

이제 이러한 기능을 JVM 및 Native에 사용할 수 있습니다. 이러한 함수에 대한 명명 규칙은 println()과 동일하며, 처음 접하는 사람에게 중요합니다.

fun main() {
    println("Input two integer numbers each on a separate line")
    val num1 = readln().toInt()
    val num2 = readln().toInt()
    println("The sum of $num1 and $num2 is ${num1 + num2}")

2. Stable Duration API​

피드백에 따라 Duration API를 안정화하고 관련 로드맵 아이템을 닫았습니다.

Duration.toString()의 보다 읽기 쉬운 출력과 1.5.30에서 미리 볼 수 있던 문자열의 날짜 변환 기능 외에도 Duration API는 다음과 같은 변경 사항이 있습니다.

  • toComponents 함수의 days 컴포넌트는 이제 Int 타입으로 인해 값이 짤리는 것을 보완하여 Long 타입을 사용합니다.
  • DurationUnit enum은 이제 type alias로 정의되어있지 않습니다. (기존에는 JVM의 java.util.concurrent.TimeUnit 의 type alias로 정의되어 있었습니다.)
  • 커뮤니티 피드백으로 Int.seconds와 같은 확장 속성을 되돌려놓았습니다. 적용 가능성을 제한하기 위해 Duration Class의 Companion에서만 사용할 수 있습니다.
val duration = 5000.seconds
println("There are ${duration.inWholeMinutes} minutes in ${duration.inWholeHours} hours")

3. Stable typeOf()​

코틀린 1.6에서 typeOf()가 안정화 버전이 되었고 관련 로드맵 항목을 닫습니다. typeOf()는 1.3.40부터 사용할 수 있었고 현재 모든 코틀린 플랫폼에서 사용할 수 있습니다. 그리고 컴파일러가 유추할 수 있는 모든 Kotlin Type을 표현하는 KType을 가져올 수 있습니다.

inline fun <reified T> renderType(): String {
    val type = typeOf<T>()
    return type.toString()
}

fun main() {
    val fromExplicitType = typeOf<Int>()
    val fromReifiedType = renderType<List<Int>>()
}

4. Stable collection builders​

코틀린 1.6.0에서 컬렉션 빌더 함수 buildMap(), buildList() 및 buildSet()가 안정화 버전이 되었습니다. Builder는 읽기 전용 Collection을 반환합니다.


5. Stable bit rotation operations for integers​

코틀린 1.6.0에서는 이진수를 지정된 비트 수만큼 왼쪽 또는 오른쪽으로 회전시키는 rotateLeft()와 rotateRight() 함수가 안정화 버전이 되었습니다.

val number: Short = 0b10001
println(number.rotateRight(2).toString(radix = 2)) // 100000000000100
println(number.rotateLeft(2).toString(radix = 2))  // 1000100

6. Stable Regex function for splitting a string into a sequence​

코틀린 1.6.0에서 문자열을 정규식으로 분할하여 sequence로 변환하는 splitToSequence()가 안정화 버전이 되었습니다.

val colorsText = "green, red , brown&blue, orange, pink&green"
val regex = "[,\\s]+".toRegex()
val mixedColor = regex.splitToSequence(colorsText)
    .onEach { println(it) }
    .firstOrNull { it.contains('&') }
println(mixedColor) // "brown&blue"

7. compareTo in infix notation​

infix 표기법으로 Comparable.compareTo 함수에서 두 개체를 비교하여 순서를 지정할 수 있는 기능이 추가되었습니다.

class WrappedText(val text: String) : Comparable<WrappedText> {
    override fun compareTo(other: WrappedText): Int =
        this.text compareTo other.text
}

8. Consistent replace() and replaceFirst() on JVM and JS​

코틀린 1.6.0 이전까지는 대체문자열에서 group reference를 참조하면 replace()와 replaceFirst() Regex 함수가 JVM과 JS에서 다르게 동작하고 있었습니다. 이제 Kotlin/JS와 JVM의 동작은 동일합니다.
