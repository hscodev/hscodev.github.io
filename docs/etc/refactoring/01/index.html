1장 맛보기 예제

리팩토링 예제 키워드​

  • 리팩토링에는 테스트가 사용된다. 코드 수정 중에 버그가 생겼는지 알기 위해 테스트 코드를 이용하자.
  • 긴 메서드를 보면 작은 부분들로 쪼갤 수 있는지 살펴보자. - 메서드 추출 기법
  • 로직이 있는 클래스 내에서 참조가 없고 다른 클래스를 참조한다면 잘못된 객체에 있는지 생각해보자 - 메서드 이동 기법
  • 임시 변수는 불필요한 중복이다. - 임시 변수를 메서드 호출로 전환 기법
    • 질의 메서드는 클래스 안의 모든 메서드에서 접근 간으하므로, 메서드를 복잡하게 만드는 임시 변수를 사용하지 않아도 되니 설계가 깔끔해진다.
  • 여기까지의 리팩토링으로 문제는 발생할 수 있다.
    • 코드가 길어진다.
    • 성능이 떨어진다.
    • 이런 이유로서 리팩토링을 하려하지 않으면 안된다. 최적화 단계가 성능 해결의 적기이며 더 많은 선택지가 생길 것이다.
  • 역시 리팩토링의 장점도 많이 생긴다.
    • 메서드를 분리함으로서 클래스 내에 어디에서든 사용할 수 있다. - 재사용성
    • 외부 시스템에서 사용하고자 하면 인터페이스를 추가하여 간단히 제공할 수 있다. - 재사용성
    • 중복코드가 없어지고 계산식 수정에는 한 곳만 수정해주면 된다. - 유지보수성
  • Switch 문의 인자는 타 객체의 속성을 사용하지 않아야 한다. 역시 적절한 클래스에 배치된 함수인지 고민해보자 - 메서드 이동 기법
  • 마지막으로 상속 구조를 만들어준다.
    • 하위 클래스를 작성하여 상속 구조를 만들면 switch문을 리팩토링 할 수 있다.
    • 디자인 패턴 중 하나인 상태 패턴을 사용할 수 있다.
    • 상태 패턴을 적용하기 위해서는 세가지 리팩토링 기법이 들어간다.
    • 분류 부호를 상태 패턴안으로 옮기는 상태/전략 패턴으로 전환 기법
    • switch문을 클래스 안으로 옮기는 메서드 이동 기법
    • switch문을 제거하는 조건문을 재정의로 전환 기법
  • 패턴을 적용하는 것은 상당히 복잡한데 적용할 가치가 있을까?
    • 패턴을 적용하게 되면 계산 방식을 변경하거나 추가동작 등을 아주 쉽게 수정할 수 잇다.
    • 프로그램의 다른 부분 역시 영향을 받지 않는다.
    • 작은 시스템에서는 미미할 지 모르지만 복잡한 시스템이라면 점점 큰 차이를 느끼게 된다.

고찰​

  • 예제의 리팩토링에서 메서드 추출, 메서드 이동, 조건문 재정의로 전환 등 리팩토링 기법을 사용하였다.
  • 이런 기법들은 기능 분배가 균등해지고 코드 유지보수도 쉬워진다.
  • 이 예제에서 알려주는 가장 중요한 교훈은 간단한 수정 → 테스트를 리듬처럼 반복해야 한다는 것이다. 이 리듬을 지킬 때만이 리팩토링을 빠르고 안정적으로 완료할 수 있다.
